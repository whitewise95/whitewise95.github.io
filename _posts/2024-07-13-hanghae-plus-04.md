# 1. 들어가는 말 

이번 과제는 아래의 API에 대한 마일스톤, 스퀀스다이어그램, ERD, API명세서를 작성하고 MockAPI를 생성하는게 이번 과제였다.

이제 마일스톤, 시퀀스다이어그램, ERD, API명세서를 작성하고 MockAPI를 생성하는 개발기를 얘기해보려고한다.


# 2. 개발기

## 📕 마일스톤 
[내가 작성한 마일스톤 자세히보기](https://github.com/users/whitewise95/projects/4/views/1)

마일스톤을 한번도 직접 작성해보지 않아서 어떤 기준으로 단위를 나눠 데드라인을 정하기 힘들었던 것 같다.
그리고 `어떤 소프트웨어를 사용해야 가독성이나 사용성이 편할까?` 라는 고민을 했다. 

팀원중에 git으로 마일스톤을 관리할 수 있다는 정보를 얻었고 가독성도 좋은 것 같고 사용성도 편한거 같아서 채택해 마일스톤을 구축해보았다.

목적이 과제제출이 아니고 직접 마일스톤을 활용해서 개발을 해보고싶다는 목적으로 작성해보았다. 

처음에는 단위를 나누고 데드라인을 정하는 과정이 까다롭게 느껴졌지만, Git을 통해 마일스톤을 관리하는 방법을 배우면서 가독성과 사용성을 동시에 확보할 수 있었고 이 과정에서 프로젝트의 진행 상황을 체계적으로 관리할 수 있는 능력을 키울 수 있었다. 

![](https://velog.velcdn.com/images/whitewise95/post/841e6b5e-bb4b-437b-92bd-d969977d4eed/image.png)

## 📕 시퀀스다이어그램
[내가 작성한 시퀀스다이어그램 자세히보기](https://github.com/whitewise95/hhplus-e-commerce/tree/main/docs)

그래도 마일스톤은 작성만 안해봤지 보기는 많이 봤지만 시퀀스다이어그램은 전혀 본적이 없어서 방향성을 잡는데 비교적 시간이 오래걸린 것 같다.

시퀀스 다이어그램이란 어떠한 순서로 어떤 객체들과 어떻게 상호작용했는지를 표현하는 것인데 직접 작성해보니, 유즈케이스를 디테일하게 알 수 있고 시나리오를 파악하기도 편했다. 
아래 이미지는 내가 직접 작성한 건 아니고 이런 느낌이다 라는 예시입니다! 
![](https://velog.velcdn.com/images/whitewise95/post/6481889a-4740-454d-a296-869b21e0cffb/image.png)

시퀀스다이어그램을 처음 접했을 때는 다소 막막했지만, 유즈케이스를 디테일하게 파악하고 시나리오를 명확히 이해하는 데 큰 도움이 되었고 객체 간의 상호작용을 시각적으로 표현함으로써 복잡한 로직을 한눈에 파악할 수 있었다. 이는 앞으로의 시스템 설계에도 큰 장점으로 작용할 것 같다.



# 📕 ERD
ERD란 사실 기획이 어떠냐에 따라 엄청 달라진다고 개인적인 생각을 가지고 있다. 
만약 자신의 어플리케이션에 히스토리를 남기지않아도 된다면 포인트적립을 하더라도 히스토리를 남기는 테이블 없을 수 있다 라는 생각이다.

그리고 전 회사에서 ERD설계는 많이 해봐서 간단한 어플리케션 ERD는 자신있게 하는 것 같다. 

어떤 API을 개발해야하고 어떤 생각을 ERD를 설계했는지 이야기 해 볼 생각이다.

### ✅ 이번 과제에서 개발해야하는 API 

1. 잔액 충전 / 조회 API
2. 상품 조회 API
2. 주문 / 결제 API
3. 상위 상품 조회 API
4. 장바구니 API


### ✅ 잔액 충전 / 조회 
잔액은 사용자에게 있어야 한다고 생각을 했기 때문에 유저 테이블을 두고 amount 라는 필드로 관리하기로 했다. 
잔액충전시 히스토리를 남기라는 요구사항이 없었기 때문에 히스토리 테이블은 따로 두지 않았다.

![](https://velog.velcdn.com/images/whitewise95/post/a2e40f2a-81a0-47d7-9d41-133d7beb3f31/image.png)


### ✅ 상품 조회 API
상품 조회를 하기 위해서는 상품테이블이 있어야하고 상품은 옵션이 없는 기획으로 ERD를 설계했다. 
![](https://velog.velcdn.com/images/whitewise95/post/d62fa137-1931-47d3-9785-3999b8fe0134/image.png)


### ✅ 주문 / 결제 API
주문시 여러 상품을 주문할 수 있기 때문에 주문상품들을 묶는 주문테이블을 두고 주문과 1 : N 관계를 가지는 주문상품 테이블 두어 관리하도록 설계했다. 

주문을 취소하거나 주문을 대기할 수 있다는 기획 없이 주문을 할 수 있다 라는 요구사항을 따라 상태를 관리하는 필드는 두지는 않았다. 

그리고 상품테이블의 상품명과 상품금액은 언제든지 수정될 수 있다는 생각에 주문상품테이블에 필드를 따로 만드는 반정규화를 진행했다. 

![](https://velog.velcdn.com/images/whitewise95/post/c7e0bec5-fee8-4f35-be94-1a011b080b47/image.png)


### ✅ 장바구니 API
장바구니는 상품의 정보가 변경되면 바로바로 최신화가 되어야 하므로 정규화로 설계 했다. 
![](https://velog.velcdn.com/images/whitewise95/post/d69e3c93-ecf3-40de-9122-a5690e5a780b/image.png)



### ✅ 전체 
[ERD 전체보기 ](https://www.erdcloud.com/d/k2Bg73mHyuFcitshf)

![](https://velog.velcdn.com/images/whitewise95/post/dc13e11a-6b85-4302-98f6-834f35c00364/image.png)


기존에 해오던 ERD 설계 경험을 바탕으로 이번 과제에서도 자신 있게 ERD를 작성할 수 있었다. 각 API의 요구사항을 바탕으로 테이블 간의 관계를 명확히 정의하고, 데이터의 정규화와 반정규화를 적절히 활용하여 효율적인 데이터베이스 구조를 설계할 수 있었다.

# 3. 마무리

이번 과제를 통해서 개발전에 설계를 하는 소중한 경험을 했다.
