# 어려웠던 점과 개선 방향
> 3개월간의 기획 빌딩 끝에 “FIX”된 기획을 기반으로 개발이 시작됐지만, 실제로는 많은 구멍이 있었다. 이번 프로젝트에서 특히 어려웠던 점은 크게 두 가지였다.

## 1) 기획의 구멍 + 잦은 변경으로 인한 재작업

기획을 외주로 맡겨 진행했고, 인계받은 기획을 기반으로 개발을 시작했다. 그런데 인계받은 기획에는 생각보다 구멍이 많았고, 그 구멍을 메우는 데에도 많은 시간이 투입됐다.

게다가 **욕심 때문에 기능이 계속 추가/변경**되면서 아래 문제가 반복적으로 발생했다.

- DB 설계 변경이 자주 발생
- DB 변경으로 인해 로직 수정이 연쇄적으로 필요
- 변경이 쌓일수록 사이드이펙트가 빈번하게 발생

특히 기획이 변경될 때마다 “사이드이펙트가 생길 만한 지점”을 전부 체크하는 일이 너무 지쳤고, 놓치는 경우도 많았다.

또한 일정은 정해져 있는데 기획은 계속 바뀌다 보니,
야근하면서 기능 개발을 끝내도 다시 기획 변경으로 엎어지는 상황이 자주 발생했다.
프로젝트 후반에는 현타가 자주 왔고, 야근에 대해서도 부정적인 감정이 커졌다.

### 미리 방지할 수 있는 방법 (내가 생각하는 개선점)

- **테스트 코드를 최대한 작성하면서 개발하기**
  - 테스트 코드가 있었으면 사이드이펙트를 막거나 최소한 빠르게 감지하는 데 도움이 됐을 것이다.
  - 결과적으로 추후 시간을 단축하는 방향이라고 느꼈다.

- **기획 문서화/흐름 정리의 중요성**
  - 기획 측면에서는 전체 흐름을 먼저 파악하고 문서화를 잘 해둔 뒤,
    변경 시 엮인 기획을 빠르게 파악할 수 있어야 한다고 생각한다.
  - 다만 이 부분은 내 파트가 아니기 때문에,
    내 포지션에서는 **문서화와 주석 처리**를 최대한 잘 해두는 게 현실적인 대응이라고 느꼈다.

## 2) 데이터 검증 비용이 과도하게 큼

개발이 완료되고 프론트가 붙는 도중, 그리고 QA 과정에서 오류처럼 보이는 데이터들이 눈에 띄었다.

그런데 테스트 과정에서 **하드로 생성한 데이터**들도 섞여 있었고,
개발계에서 내부 운영 테스트를 하다 보니 “왜 데이터가 이렇게 나왔는지”를 검증하는 데 시간이 너무 오래 걸렸다.

### 개선 방향

- 개발계에서 테스트하더라도 하드로 넣은 값보다는,
  **최대한 정확하고 실제 케이스에 가까운 데이터**를 넣어두는 것이 좋겠다.

- 프론트/QA에서 테스트 데이터를 요청할 때마다 즉흥적으로 넣다 보면
  결국 “테스트할 수 있는 데이터만 대충 넣어두는 상황”이 생긴다.

- 필요한 쿼리는 시간이 들더라도 **한번 잘 만들어두고 재사용**하는 것이
  장기적으로 시간을 단축하는 방법이라고 느꼈다.

### 결론: 바쁠수록 더 정리해야 한다

예전에 다른 직종에서 일할 때 들었던 “일에는 순서가 있다”는 말이 계속 떠올랐다.

모든 일을 할 때 바쁘다는 핑계로 중요한 정리나 준비를 건너뛰기보다는,
**바쁘기 때문에 더 정리하고, 더 투자해야 후일을 도모할 수 있다**는 걸 이번 프로젝트에서 크게 느꼈다.

# 고민했던 내용 그리고 선택
## 1) “공정함”을 만들기 위해 고민했던 방법들,
> 특정 시간에 대회에 참가한 유저들을 대상으로, 발행 비율에 맞춰 경품을 지급하는 기능이 있었다. 
>
>이 기능은 배치 Job으로 구현되어 정해진 시간에 실행되도록 구성했다.  
>
> 다만 발행 대상 유저를 조회하는 로직이 청크 단위(1,000명)로 끊어서 처리되면서, 참가자가 1,000명을 초과하는 경우 청크 간 순서/대상 편차가 생겨 공정성이 어긋날 수 있는 문제가 있었다.

### 문제 정의: “랜덤”이 아니라 “공정함”이 필요했다
이번 작업(보상/선정/처리 순서 결정)에서 핵심은 단순히 **무작위로 섞는 것**이 아니라,

- 특정 유저가 계속 먼저 처리되거나
- 특정 구간(낮은 ID, 최근 가입자, 특정 정렬 조건)이 반복적으로 유리해지거나
- 실행 시점마다 결과가 예측 가능해져 조작/의심 여지가 생기는 것

이런 “편향”을 줄여서 **실행마다 공정하게 기회를 분산**시키는 것이 목표였다.

### 공정함을 위해 실제로 고민했던 접근들

#### A. 기본 정렬(예: `ORDER BY account_id`, `created_at`)
- 장점: 빠르고 안정적, 인덱스도 잘 탐
- 단점(치명): 실행할 때마다 **같은 사람이 계속 앞순서**를 가져감  
  → “공정함”의 목적과 정면 충돌

결론: 성능은 좋은데 공정성 요구를 충족 못 함.

---

#### B. `ORDER BY random()` 같은 DB 랜덤
- 장점: 진짜 랜덤, 구현이 매우 간단
- 단점: 대부분의 DB에서 **후보 row 전체를 랜덤값 생성 + 정렬**해야 해서 매우 비쌈  
  → 데이터가 늘면 바로 병목

결론: 공정성은 좋지만 비용이 과도하고 운영 리스크가 큼.

---

#### C. “라운드로빈/커서 기반” 공정 분배 (지난번 마지막 처리 지점부터 이어가기)
- 예시: `last_processed_account_id`를 저장해두고 다음 실행은 그 뒤부터 시작
- 장점: 특정 유저가 계속 앞에 오는 문제를 구조적으로 해결 (정렬은 유지하면서 공정해짐)
- 단점:  
  - 중간에 탈퇴/추가/필터 조건 변화가 생기면 **순환 논리가 깨지기 쉬움**
  - 조건이 여러 개면(챌린지별, 그룹별, 상태별) “마지막 지점”을 저장/관리할 키가 늘어나 복잡해짐
  - 특정 구간이 비는 경우(필터 결과가 듬성듬성) 처리 공백이 생길 수 있음

결론: 가장 “정석적인 공정성”에 가깝지만, 구현/운영 복잡도가 급상승.

---

#### D. 사전 셔플 테이블/셔플 키 컬럼(미리 랜덤 키를 저장하고 그걸로 정렬)
- 장점: 정렬 키가 컬럼이므로 인덱싱/튜닝 여지가 생김
- 단점:
  - 셔플 키를 갱신하는 주기/조건을 또 설계해야 함
  - “실행마다 공정하게 섞인다”를 만족하려면 결국 갱신 작업이 필요
  - 데이터 정합성/동시성까지 고려하면 생각보다 설계가 커짐

결론: 성능/공정성 균형은 좋은데 “당장” 적용하기엔 작업 범위가 커짐.

---

#### E. 애플리케이션에서 셔플/샘플링(조회는 정렬 없이 하고, 앱에서 섞기)
- 장점: DB 정렬 부담을 줄이고 로직 유연성이 큼
- 단점:
  - 한 번에 큰 후보를 메모리로 가져오면 애플리케이션이 부담
  - 배치/멀티 인스턴스 환경이면 **재현성/중복 처리**를 맞추기 어려움
  - “실행 시점 기준 동일 결과” 같은 요구가 있으면 별도 seed 관리가 필요

결론: 가능하지만, 운영 환경/중복 처리 정책까지 같이 결정해야 함.

---

### 그래서 `md5`가 “현실적인 절충안”이 됐던 이유

여기서 우리가 원했던 건 사실 “완벽한 랜덤”이 아니라 아래 3가지였다.

1) **편향이 적을 것** (항상 같은 사람이 먼저 오지 않게)  
2) **재현 가능할 것** (같은 runAt이면 같은 결과 → 디버깅/검증 가능)  
3) **구현 비용이 작을 것** (설계 확장 없이 당장 적용 가능)

이 조건을 놓고 보면 `ORDER BY md5(account_id || runAt)`은 절충점이 됐다.

- `account_id`만으로 정렬하면 편향이 고정되는데  
  → `runAt`을 섞으면 실행마다 순서가 바뀜 (편향 완화)
- 완전 랜덤(`random()`)은 매번 결과가 달라 검증이 어려운데  
  → `md5`는 “입력값이 같으면 항상 같은 결과”라 재현성 확보
- 별도 테이블/상태 저장/복잡한 분배 로직 없이  
  → SQL 한 줄로 “공정해 보이는 셔플”을 만들 수 있음 (적용 비용 최소)

즉, **공정성 + 재현성 + 낮은 구현비용**을 동시에 만족하는 가장 빠른 선택지가 `md5`였다.

---

### 선택의 트레이드오프도 분명했다
`md5`는 “공정함을 빠르게 달성하는 방법”이었지 “가장 성능 좋은 방법”은 아니었다.

- 후보가 커지면 결국 `md5 계산 + 전체 정렬` 비용이 듦
- 규모가 커지면 “셔플 키 컬럼화”나 “라운드로빈” 같은 구조적 해결이 더 적합해짐

하지만 그 시점의 우리는,
- 공정성 요구를 빨리 충족해야 했고
- 운영에서 검증 가능한 재현성이 필요했고
- 설계 변경을 크게 하기 어렵거나(일정/리스크/영향 범위) 우선 MVP로 가야 했기 때문에

**“지금 당장 가장 합리적인 절충안”으로 `ORDER BY md5`를 선택했다.**


## 2) "웹소켓 인가"는 어디까지 해야 하나?

> 처음에는 `@MessageMapping("/chat.send")` 같은 메서드에서도 매번 인가(Authorization) 검사를 넣는 게 가장 안전하다고 생각했다.  
하지만 WebSocket은 HTTP처럼 요청-응답이 끝나는 구조가 아니라 연결이 유지되는 실시간 채널이다. 메시지마다 인가 검사를 반복하면 지연이 늘고, 코드가 불필요하게 복잡해진다는 걸 알게 됐다.

### 문제 정의: “매 메시지 인가”는 안전하지만, 실시간성과 유지보수성을 해친다
당시 내가 우려했던 건 두 가지였다.

- 누군가 세션을 탈취하거나 토큰을 위조해 메시지를 보내는 상황
- 특정 엔드포인트(예: 채팅 전송)에서 권한 체크가 누락되어 보안 구멍이 생기는 상황

그래서 처음에는 “메시지를 받을 때마다 인가 체크를 하면 가장 안전하다”라고 판단했다.  
하지만 WebSocket의 특성상 이 방식은 다음 문제를 만들었다.

- 메시지마다 인증/인가 로직이 실행되어 레이턴시가 증가
- 핸들러(@MessageMapping)마다 중복 코드가 늘어 복잡도 상승
- 실시간 트래픽이 늘어날수록 서버 부하가 선형적으로 커짐

즉, 보안은 강화되지만 실시간성이라는 WebSocket의 장점을 깎아먹는 구조였다.

---

### 인가를 위해 실제로 고민했던 접근들

#### A. `@MessageMapping` 메서드에서 매번 인가(Authorization) 검사
- 장점: 가장 직관적이고 안전해 보인다
- 단점: 메시지 단위 비용이 커지고 중복 코드가 많아진다  
  → 실시간 처리에서 성능/복잡도 문제가 빠르게 누적된다

결론: 안전하지만, 운영 규모를 고려하면 유지하기 어려운 구조였다.


#### B. 핸드셰이크(HTTP Upgrade) 시점에 `Authorization` 헤더로 처리
- 장점: HTTP처럼 “처음 한 번만 검증”하면 되니 깔끔하다
- 단점: 브라우저 환경에서는 핸드셰이크 요청에 임의 헤더(Authorization)를 자유롭게 넣기 어렵다  
  → 이상적이지만 프론트/브라우저 제약 때문에 현실적으로 적용이 까다롭다

결론: 개념은 가장 깔끔했지만, 구현 현실과 충돌했다.


#### C. CONNECT 시점(STOMP 헤더)에서만 인가하고 세션을 신뢰
- 장점: 인가를 “연결 단계”에 집중시키고, 이후 메시지는 가볍게 처리할 수 있다  
  → 실시간성과 코드 단순성을 동시에 확보
- 단점: “연결 후 세션을 신뢰한다”는 전제가 생기므로, 세션 관리/만료/차단 정책이 중요해진다  
  → Redis 세션 상태가 꼬이거나 만료 전략이 부실하면 보안/정합성 문제가 생길 수 있다

결론: WebSocket 환경에서 현실적인 균형점이었다.


### 그래서 CONNECT 시점 인가가 “현실적인 절충안”이 됐던 이유
우리가 원했던 건 “매 메시지마다 무조건 검사”가 아니라, 아래 3가지였다.

1) 연결을 열 때 확실히 검증할 것  
2) 메시지 처리 경로는 가볍게 유지할 것(실시간성)  
3) 운영에서 추적/차단이 가능할 것(세션 식별)

그래서 최종적으로 다음 흐름으로 정리했다.

- CONNECT 시점에 STOMP 헤더로 JWT를 전달받는다
- JWT 파싱/검증을 수행한다
- Redis에 세션 정보를 저장한다 (`accountId`, `consumerId`, `sessionId` 등)
- 이후 `send` 단계에서는 “이미 인증된 세션”을 전제로 인가 검사를 반복하지 않는다

즉, 보안을 “연결 단계에서 확실히” 하고, 메시지 경로는 “가볍게” 만들었다.


## 3) Kafka에 저장된 메시지를 어떻게 정확히 해당 유저 세션(WebSocket)으로 전송할 것인가?

> 채팅 서버는 총 6대의 VM으로 구성될 예정이었다.  
> 이때 가장 큰 질문은 하나였다.  
> “Kafka에 저장된 메시지를 어떻게 **해당 유저의 WebSocket 세션이 붙어있는 VM**으로 정확히 전달할 것인가?”

---

### 문제 정의: 브로드캐스트가 아니라 “정확한 라우팅”이 필요했다
채팅 메시지는 모든 인스턴스가 동일하게 받는 구조가 아니라, **특정 유저(또는 특정 세션)에게만 전달**되어야 했다.

- Kafka에는 메시지가 쌓이지만, “어느 VM이 어느 유저의 세션을 가지고 있는지”는 Kafka가 알 수 없다
- 모든 VM이 모든 메시지를 컨슘하면, 중복 전송/불필요한 처리/비용이 폭발한다
- 결국 메시지를 컨슘한 VM이 **해당 유저 세션을 실제로 보유하고 있어야만** WebSocket으로 전송이 가능하다

즉, 목표는 “처리 분산”이 아니라 **세션 소유자에게 정확히 전달하는 라우팅**이었다.

---

### 초반에 고민했던 대안들

#### A. Redis Pub/Sub
- 장점: 구조가 단순하고 빠르게 붙일 수 있다
- 단점:
  - 메시지 유실 가능성이 있으며 재처리가 어렵다
  - 장애 복구가 까다롭다
  - 인스턴스/채널이 늘수록 Redis 부담이 커진다

결론: 단기 구현은 쉽지만 운영 안정성이 약했다.

---

#### B. gRPC 기반 인스턴스 간 메시지 브로드캐스트
- 장점: 모든 인스턴스에 직접 전달하는 구조를 만들 수 있다
- 단점:
  - 인스턴스 수가 늘면 연결/전송/재시도 관리가 기하급수로 복잡해진다
  - 장애 상황에서 누락/중복/순서 보장 같은 운영 이슈가 커진다

결론: 구현은 가능하지만 운영 복잡도가 높아 장기적으로 부담이 컸다.

---

### 해결 방향: “파티션은 VM에 고정”, “세션 소유자는 Redis로 추적”
최종적으로 선택한 방식의 핵심 아이디어는 두 가지였다.

1) **Kafka 파티션을 VM별로 고정 분배**해 “이 파티션은 이 VM이 처리한다”를 명확히 한다  
2) Redis에 `accountId → consumerId(VM)`를 저장해 “이 유저는 현재 어느 VM에 붙어있다”를 추적한다

이를 통해 메시지 흐름은 다음처럼 정리됐다.

- 유저가 WebSocket CONNECT 시점에 접속  
  → Redis에 `accountId → consumerId` 저장
- 메시지를 Kafka에 publish 할 때  
  → `accountId`를 키로 사용해 “특정 파티션”으로 들어가게 구성(키 기반 라우팅)
- 해당 파티션을 담당하는 VM이 메시지를 consume  
  → 자신이 보유한 WebSocket 세션으로 전송

즉, **Kafka는 VM 단위 분산**, **Redis는 세션 위치 라우팅**, **WebSocket은 최종 전달** 역할을 맡는 구조가 되었다.

---

### 파티션 분산 처리 전략: VM 환경변수 기반 컨슈머 설정
각 VM은 실행 시 아래 환경변수를 주입받는다.

```bash
# VM1
-Dkafka.chat.consumer.id=chat-consumer-01
-Dkafka.chat.consumer.total-consumer-count=3

# VM2
-Dkafka.chat.consumer.id=chat-consumer-02
-Dkafka.chat.consumer.total-consumer-count=3

# VM3
-Dkafka.chat.consumer.id=chat-consumer-03
-Dkafka.chat.consumer.total-consumer-count=3
```

이 값은 Spring Boot 설정으로 주입된다.
```yml
kafka:
  chat:
    consumer:
      id: ${CONSUMER_ID}
      total-consumer-count: ${TOTAL_CONSUMER_COUNT}
```

컨슈머 ID는 `consumerIndex`를 계산하는 데 사용했다.  
인덱스는 0부터 시작하므로 `chat-consumer-01 → 0`, `chat-consumer-02 → 1`처럼 매핑된다.

```java
public class KafkaChatConsumerProperty {

    private String id;
    private Integer totalConsumerCount;

    public Integer getConsumerIndex() {
        String[] splitArr = this.id.split("-");
        return Integer.parseInt(splitArr[splitArr.length - 1]) - 1;
    }
}
```

### 파티션 분산 원리: `partition % totalConsumerCount == consumerIndex`
```java
public void startPartitionedListener() {

    TopicPartitionOffset[] offsets =
        IntStream.range(0, getPartitionCount(getTopicName(TOPIC))) // 토픽의 총 파티션 수
                 .filter(p -> p % kafkaChatConsumerProperty.getTotalConsumerCount() == CONSUMER_INDEX)
                 .mapToObj(p -> new TopicPartitionOffset(TOPIC, p))
                 .toArray(TopicPartitionOffset[]::new);

    // ... offsets 기반으로 listener/container 구성
}
```
각 VM은 “내가 담당할 파티션”만 직접 필터링해서 리스너를 구성했다.

예를 들어 토픽 파티션이 6개이고, 총 컨슈머 수가 3이며 현재 VM 인덱스가 0이라면:

- 파티션: `[0, 1, 2, 3, 4, 5]`
- 조건: `p % 3 == 0`

따라서 `chat-consumer-01`은 파티션 `0, 3`을 담당하게 된다.  
이 방식으로 모든 인스턴스는 서로 겹치지 않고, 정해진 파티션만 고정 처리한다.


### 이 방식의 장점: 예측 가능성과 확장성
이 구조의 장점은 운영 관점에서 특히 컸다.

- 각 VM이 담당할 Kafka 파티션이 고정되어 컨슘 범위가 예측 가능하다
- Redis에 `accountId → consumerId`만 저장하면 메시지를 정확히 해당 VM으로 라우팅할 수 있다
- 인스턴스를 늘리거나 줄일 때, 환경변수만 조정해 파티션 분배를 재구성할 수 있다
- VM을 수평 확장할수록 처리량을 확장할 수 있어 구조적으로 확장성이 좋다

결국 “모든 메시지를 모든 인스턴스가 처리”하는 방식이 아니라,  
“세션을 가진 VM만 메시지를 처리하고 전송”하도록 만들어 비용과 복잡도를 함께 줄일 수 있었다.

# 이슈
## 1) DB 인코딩/Collation(locale) 변경 이후 문자열 컬럼 비교가 실패한 이슈

인코딩 또는 Collation(locale) 변경 이후, **인덱스가 걸려있는 문자열 컬럼**에서 비교가 기대대로 동작하지 않는 문제가 발생할 수 있다.

대표 증상은 다음과 같다.

- `=`, `IN`, `JOIN ON`, `WHERE col = '값'` 같은 비교 조건이 매칭되지 않는다
- 값이 같은 것처럼 보이는데도 조회 결과가 0건으로 나온다
- 인덱스가 걸린 컬럼에서 비교/조인이 에러가 나기도 한다
- (DB에 따라) 인덱스를 사용하지 못하고 실행 계획이 비정상적으로 바뀌거나,
  비교 자체가 거부되는 경우도 있다

---

Collation/encoding 변경 이후 가장 흔한 해결책은 **관련 인덱스를 재생성**하는 것이다.  
특히 문자열 관련 인덱스(일반/유니크 인덱스)는 영향도가 크다.

예를 들어, 스키마 단위로 재생성이 필요하다면 다음과 같이 수행할 수 있다.

```sql
REINDEX SCHEMA comm;
```

## 2) STOMP Receipt 도입 금지어 체크 동기 응답)

### 배경
채팅 메시지를 STOMP로 전송하는 구조에서, 사용자가 메시지를 보냈을 때 **금지어가 포함되면 즉시 실패를 알려줘야** 했다.  
HTTP 요청처럼 “요청 → 응답”이 자연스럽게 보이지만, STOMP `SEND`는 기본적으로 **fire-and-forget** 성격이라 프론트가 “서버가 이 메시지를 정상 처리했는지/거부했는지”를 즉시 알기 어려웠다.

특히 아래 요구가 핵심이었다.

- 사용자가 채팅 입력 후 전송했을 때, **금지어면 곧바로 에러 토스트/안내**를 띄우고 싶다.
- 서버가 금지어를 감지하면 메시지 저장/브로드캐스트를 **원천 차단**해야 한다.
- 프론트는 “내가 보낸 이 메시지”가 실패한 것인지 **정확히 매칭**해야 한다.


### 문제: STOMP의 SEND는 동기 응답이 없다
STOMP `SEND`는 요청-응답 구조가 아니어서, 서버에서 검증 실패가 나도 프론트는 보통 다음 중 하나로만 감지한다.

- 메시지가 상대방에게 안 갔다(간접적으로 추정)
- 서버 로그/모니터링에서만 확인 가능
- 별도의 에러 채널을 직접 설계해야 한다

즉, “금지어 실패”를 유저 경험 상 **즉시 피드백**하려면, STOMP 흐름 위에 **동기 응답처럼 보이는 메커니즘**을 얹어야 했다.


### 선택: ReceiptId 기반으로 ‘요청-응답처럼’ 만들기
결론적으로 프론트가 메시지를 보낼 때 `ReceiptId`(또는 표준 `receipt`)를 같이 보내고,  
서버는 검증 결과를 해당 `ReceiptId`와 함께 되돌려서 프론트가 “내 요청에 대한 응답”으로 처리하도록 했다.

#### 의도
- 프론트: `ReceiptId`로 pending 상태를 잡아두고, 응답이 오면 성공/실패를 확정
- 서버: 금지어면 메시지를 drop하고(저장/브로드캐스트 안 함), **응답 프레임**을 보내서 즉시 실패를 전달


### 구현: BanWordChannelInterceptor에서 차단 + 응답 전송
현재 구현은 `ChannelInterceptor#preSend`에서 다음을 수행한다.

1. `SEND` 프레임인지 확인
2. destination이 `/app/chats/send` 인지 확인
3. payload를 문자열로 파싱
4. 금지어 포함 여부 검사
5. 금지어면:
   - 프론트가 보낸 `ReceiptId`를 읽는다
   - `{ code, status, error, message(banWords) }` 형태의 에러 바디를 구성한다
   - `RECEIPT` 프레임을 만들어 `receipt-id`에 `ReceiptId`를 세팅한다
   - `clientOutboundChannel`로 전송하고 원본 메시지는 `return null`로 막는다

```java
if (banWordUtils.containsBanWords(payload)) {
    List<String> banWords = banWordUtils.getBanWords(payload);

    String receiptId = Objects.requireNonNull(accessor.getNativeHeader("ReceiptId"))
        .stream().findAny().orElse(null);

    if (receiptId != null) {
        Map<String, Object> response = new HashMap<>();
        response.put("code", "COMMON_400_0002");
        response.put("status", 400);
        response.put("error", "BAN_WORD");
        response.put("message", banWords);

        StompHeaderAccessor out = StompHeaderAccessor.create(StompCommand.RECEIPT);
        out.setSessionId(accessor.getSessionId());
        out.setReceiptId(receiptId);

        outboundProvider.getIfAvailable()
            .send(MessageBuilder.withPayload(objectMapper.writeValueAsBytes(response))
                .setHeaders(out)
                .build());

        return null; // 원본 SEND 차단
    }
}
```

### 얻은 효과

- **즉시 피드백**  
  사용자는 전송 직후 금지어 안내를 바로 받는다.

- **정확한 매칭**  
  `ReceiptId`로 “어떤 메시지가 실패했는지” 프론트가 확정 가능하다.

- **서버 보호**  
  금지어 메시지는 저장/브로드캐스트 전에 `preSend` 단계에서 차단된다.

---

### 트레이드오프 & 아쉬운 점

#### `RECEIPT`를 ‘에러 응답’ 용도로 사용한 점
`RECEIPT`는 본래 “서버가 해당 프레임을 처리했다”는 확인(ack) 성격인데,  
여기에 에러 바디를 실어 “비즈니스 에러 응답”으로 쓰면서 의미가 섞였다.

- **장점**: 구현이 빠르고 프론트가 `receipt-id` 기반으로 쉽게 연결 가능  
- **단점**: STOMP 의미론과 살짝 어긋나고, 다른 클라이언트/라이브러리에서 기대와 달라질 수 있다

#### 헤더 강제 접근(NPE 위험)
`Objects.requireNonNull(accessor.getNativeHeader("ReceiptId"))`는 헤더가 없으면 NPE가 발생할 수 있다.  
즉 “금지어 실패”가 아니라 “서버 오류”로 번질 여지가 있다.

#### outboundChannel 직접 전송의 결합도
`clientOutboundChannel`로 직접 프레임을 쏘는 방식은 강력하지만,  
Spring 메시징 추상화(`SimpMessagingTemplate`, `@SendToUser`) 대비 결합도가 높아질 수 있다.

---

### 다음 개선 아이디어

- `ReceiptId`를 커스텀(`ReceiptId`) 대신 **표준 `receipt` 헤더**로 통일해서 호환성을 높이기

- 응답 채널을 `RECEIPT`에 바디를 싣는 대신,
  - `/user/queue/errors` 같은 **에러 전용 destination**으로 보내고
  - `receiptId`로 **상관관계만 유지**하는 구조로 정리하기

- `ReceiptId` 없을 때의 **fallback 정책**(무응답/일반 에러 채널/로그)을 명확히 하기

- 타임아웃/재시도 정책을 프론트와 합의해서 **“pending이 영원히 남는 문제”**를 방지하기

---

### 정리

이번 금지어 체크 이슈를 해결하면서 깨달은 건,  
**실시간 메시징에서도 결국 사용자는 동기 응답 경험을 원한다**는 점이었다.

STOMP는 기본적으로 비동기 전송이지만,  
`ReceiptId(receipt)`를 상관관계 키로 사용하면  
**“요청-응답처럼 보이는 사용자 경험”**을 만들 수 있었다.

다만 `RECEIPT`의 의미를 확장해 에러 바디까지 실어 쓰는 선택은 빠르게 문제를 해결해줬지만,  
장기적으로는 **표준 흐름(에러 채널 분리, receipt 표준화)**로 정리하는 것이 유지보수에 더 유리하다는 결론을 남겼다.

# 프로젝트 후기

출시까지 달리면서 느낀 건, “기능을 만드는 것”보다 **운영 가능한 품질로 다듬는 것**이 훨씬 오래 걸린다는 점이었다.  
특히 일정이 촉박할수록 “일단 되게 만들자”는 선택을 하기 쉬운데, 그 선택들이 프로젝트 후반부에 **성능 이슈·예외 케이스·데이터 정합성 문제**로 되돌아왔다.

## 1) 성능: 조인에 대한 과도한 의존이 슬로우쿼리로 이어졌다

이번 프로젝트에서는 요구사항을 빠르게 충족시키기 위해 여러 도메인을 한 번에 조회하려다 보니,  
쿼리가 자연스럽게 **조인 중심**으로 커지는 경우가 많았다.

- 화면/기능 단위로 “필요한 데이터를 한 번에 가져오자”는 접근이 반복됨
- 도메인 간 관계가 복잡해질수록 조인이 누적되어 쿼리가 비대해짐
- 인덱스가 있어도 조인 순서/카디널리티에 따라 실행계획이 흔들리면서 슬로우쿼리로 연결됨

결과적으로 프로젝트 후반으로 갈수록 “조회는 되는데 느리다”가 자주 발생했고,  
그때마다 운영/QA에서 병목을 찾아 튜닝하는 데 시간을 많이 썼다.

### 배운 점: 조회의 목적에 따라 `JOIN` 대신 `EXISTS` / 서브쿼리가 더 나을 수 있다
특히 “어떤 데이터가 **존재하는지**만 확인하면 되는 케이스”에서 조인은 불필요하게 비용이 커졌다.

- 조인은 결과 row 수를 불필요하게 늘리거나(중복), 정렬/그룹핑 비용을 유발할 수 있음
- 반면 `EXISTS`는 **존재 여부만 확인하고 빠르게 short-circuit** 될 수 있어 실행계획이 더 유리해지는 경우가 있었다

이번 프로젝트에서도 “조인으로 끌고 오던 조건”을 `EXISTS` 서브쿼리로 분리하면서  
슬로우쿼리가 의미 있게 줄어든 경험이 있었고, 이후로는 조회 전략을 더 신중하게 생각하게 됐다.

> 결론적으로, “데이터를 가져오는 목적”을 먼저 정의하고  
> **JOIN/EXISTS/서브쿼리/분리 조회** 중 어떤 방식이 맞는지 선택하는 습관이 중요하다는 걸 배웠다.

---

## 2) 트러블슈팅은 정말 많았는데, 기록을 남기지 않아 아쉽다

프로젝트 진행 중에는 생각보다 다양한 이슈가 계속 터졌다.

- 성능 이슈, 정합성 이슈, 인프라/환경 이슈, 배치/동시성 이슈
- WebSocket/Kafka/Redis/DB 레벨에서의 예외 상황
- QA 과정에서만 드러나는 “엣지 케이스”들

그때그때 해결하느라 정신이 없어서 “일단 해결”에 집중했고,  
결과적으로 트러블슈팅 내용을 따로 구조적으로 기록하지 못했다.

그래서 지금 회고를 쓰는 시점에서는  
**어떤 문제가 있었고, 어떤 가설로 접근했고, 어떤 결론으로 해결했는지**를  
세세하게 다 떠올리기 어렵다는 점이 가장 아쉽다.

> 다음에는 이슈가 발생했을 때 “해결”만이 아니라,  
> **문제/원인/해결/재발 방지**를 최소한의 템플릿이라도 남기는 습관을 가져가고 싶다.

---

## 3) 코드 품질: 가독성과 유지보수까지 챙기는 코드를 못 쓴 부분이 아쉽다

출시 일정에 맞추기 위해 기능 개발 속도를 우선하면서,  
코드가 점점 “빨리 만들기 좋은 형태”로 누적되는 순간들이 있었다.

- 책임이 과도하게 커진 서비스/배치 로직
- 도메인 규칙이 여기저기 흩어진 코드
- 테스트가 부족해 리팩토링이 두려운 상태
- 시간이 지나면 다시 읽기 힘든 조건문/쿼리/핵심 로직

돌아보면 “작동하는 코드”는 만들었지만,  
**읽기 쉽고 유지보수에 유리한 코드**를 끝까지 챙기지 못한 부분이 분명히 있었다.

특히 후반으로 갈수록 변경이 잦았기 때문에  
가독성/구조/테스트가 부족한 코드는 수정 비용을 더 크게 만들었고,  
그게 다시 야근/피로로 이어지는 악순환도 있었다.

> 다음 프로젝트에서는 “기능 구현”과 동시에  
> **테스트 + 구조 + 네이밍 + 책임 분리**를 최소 기준으로 잡고,  
> 속도와 품질 사이의 균형을 더 잘 맞추고 싶다.

# 마무리

결국 이번 프로젝트는 단순히 서비스를 출시한 경험을 넘어,  
“운영 가능한 시스템”을 만들기 위해 무엇이 필요한지 몸으로 배운 시간이었다.

- 공정성/재현성 같은 제품 요구를 기술적으로 풀어내는 과정
- 실시간 시스템에서의 라우팅/인가/동기 응답 경험 설계
- 성능 이슈가 설계 선택(JOIN 의존)과 얼마나 직결되는지
- 트러블슈팅 기록과 코드 품질이 장기적으로 시간을 절약해준다는 사실

많이 부딪혔고 아쉬움도 남지만,  
그만큼 다음 프로젝트에서는 더 단단한 기준과 습관을 가지고 시작할 수 있을 것 같다.
