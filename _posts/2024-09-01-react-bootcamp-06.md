# 1. 시작
6월 중순쯤 아웃소싱 프로젝트가 끝난 후, 한 달 뒤에 있을 최종 프로젝트를 위한 팀 빌딩이 시작되었다.

아웃소싱프로젝트 때 같이 팀원이었던 한 분이 최종프로젝트 때 같은 팀을 하지 않겠냐는 DM이 와서 최종프로젝트는 내가 원하는 팀원들과 같은 팀을 이루어 시작할 수 있었다. 

7/16 ~ 8/21 거의 한달 동안 힘들고 즐거웠던 최종프로젝트가 마무리되었다.
오늘은 최종프로젝트하는 동안 있었던 이야기를 해 볼 생각이다. 


 
# 2. 최종프로젝트
각자 원하는 서비스를 생각해보고 만나자고 저녁에 만나기로 했다. 
많은 의견들이 나왔지만 큼직큼직한 의견은 아래 4개로 좁혀졌다.

1. 원하는 메모해 모아두는 서비스
2. 여행 정보 공유 서비스
3. 목표를 세워 달성하는 서비스
4. 반련견을 위한 서비스

투표를 했고 `여행 정보 공유 서비스`와 `목표를 세워 달성하는 서비스`가 막상막하 였고 `나는 목표를 세워 달성하는 서비스에 한표` 를 주어 `목표를 세워 달성하는 서비스를 기획`하게 되었다. 


## 기획

`🙍‍♀️ "목표를 세우지만, 이를 체계적으로 관리하지 못하거나 중도에 포기 해요"`  
`🙎‍♂️ "너무 많은 목표를 한꺼번에 세워서 집중하기가 어려워요."`  
`👨‍✈️ "목표를 잊어버리거나 우선순위를 놓치곤 해요."`  

이러한 문제를 해결할 수 있도록 설계된 목표 관리 플랫폼으로 사용자가 목표를 구체적으로 설정하고, 계획을 세밀하게 수립하며, `지속적으로 실행할 수 있도록 돕고 싶다는 의도` 로 기획된 서비스다.

해당 서비스를 생각한 팀원은 기능 하나하나에 정말 구체적인 구상을 하고 있어서 너무 놀라기 했지만 반대로 너무 많은 기능을 생각하고 있었어서 불안하기도 했다.

수요일부터 시작된 기획은 목요일에 오후에 끝났고 이후 디자이너님께 정리된 기획을 전달하여 와이어프레임이 나올 때 까지 기다리고 있었다.

일요일이 다 지나갈 때 쯤 디자이너님께 긴 DM이 왔다. 
정리하면, 현재 일을 병행하고 있는데 작업량이 많아 피해만 드릴 것 같아서 하차한다는 소식이였다.

내배캠 매니저님들도 비상이였던거 같다. 다행이게도 디자이너분이 더 있으셨던거 같다.
팀원중 한분이 받은 DM을 공유받았다.

## 트러블
이렇게 부트캠프로 최종 팀프로젝트를 여러 번 해봤지만 프로젝트를 할 때마다 실력을 둘째치고 인생을 배워가는 것 같다. 

기획이 이미 디자이너님께 FIX에서 전달했지만 개발자 팀원들끼리 의논할 때 여러번 바뀌거나 추가하거나 하는 경우가 많이 생겼다. 

내 생각에는 디자이너님도 이미 받은 기획안을 보면서 이미 구상을 하고 있을 것 같고 더 이상 기획을 변경하거나 스케일을 높히면 오히려 산으로 갈 것 같다는 생각에, 기획에 대한 내용을 계속 말하는 팀원에게 더 이상은 기획이 추가되거나 변경되면 안될 것 같다. 라는 강한 의견을 전달하게 되었다.

아마... 이번 프로젝트는 중간에 디자이너님도 바뀌고 구현해야할 기능과 웹퍼블의 난이도도 높다고 생각하고 더불어 인원도 다른 팀에 비해 1~2명 정도 적었기에 예민해져 있었던거 같다.

### 해결방안
유튜브에 댓글을 남겨보면 refetch가 되기전까지는 인기순이든 최신순이든 내가 작성한 글이 맨 위로 올라온다.

직접뜯어본건 아니지만 뭔가 fetch해서 가져온 댓글 영역과 생성한 댓글에 영역이 다르다는 느낌을 받았고 거기에 아이디어를 얻었다.

예를 들면 아래 이미지에서 빨간 네모안은 새로 작성한 글 영역이고 파란부분은 처음 fetch해서 가져온 부분인 것이다.

다시 새로고침을 하던가 언마운트되고 다시 렌더링된다면 빨간 영역은 어떤 댓글도 없을 것이고 파란영역이 3개의 댓글로 되는 것이다.

## 알림기능

우리 서비스는 챌린지를 생성하고 종료되기 3일전에 알림을 주자는 기획이 존재했다.

그래서 종료가 3일 남은 진행중인 모든 챌린지 대상으로 조회하고 해당 유저들에게 알림을 보내주는 기능이 필요한데, 자바와 스프링으로 진행하면 스케줄러나 배치를 사용해 러프하게 잡아도 2시간이 되는 기능인데 거의 이틀을 고생했다.

### 첫 방향성
[node-schedule](https://www.npmjs.com/package/node-schedule) 라이브러리를 이용하는 것이다. 

이 라이브러리는 특정 날짜에 실행되도록 작업(임의의 함수)을 스케줄링할 수 있다.

아래 처럼 `*****` 로 설정하면 오전 12시에 함수가 실행된다.

```ts
schedule.scheduleJob("* * * * *", async function () {

  await supabase
    .from("log_scheduler")
    .insert([{ message: "스케줄러 시작"}])


  const pgClient = createPGClient()

  try {
    await pgClient.connect()

    const newVar = await pgClient.query(
      "SELECT DATE_PART('day', DATE_TRUNC('day', m.end_at) - DATE_TRUNC('day', now())) + 1 AS days_remaining, m.end_at::date AS end_date, u.nickname, c.user_id, c.id ,c.goal FROM milestone m INNER JOIN challenge c ON c.id = m.challenge_id INNER JOIN users u ON u.id = c.user_id WHERE DATE_TRUNC('day', m.end_at) BETWEEN DATE_TRUNC('day', now()) AND DATE_TRUNC('day', now()) + INTERVAL '3 days' AND m.start_at <= now() AND m.end_at >= now();"
    )

    const dataList = newVar.rows

    for (const dataListElement of dataList) {
      const { error } = await supabase.from("users_notice").insert([
        {
          challenge_id: dataListElement.id,
          user_id: dataListElement.user_id,
          content: `${dataListElement.nickname} 님  ${dataListElement.goal} 챌린지의 다음 마일스톤을 수정 및 생성할 수 있는 기간이  ${dataListElement.days_remaining}일 남았습니다.`,
        },
      ]).select()

      if (error) {
        throw new Error(error.message)
      }
    }
  } catch (error: any) {
    await supabase
      .from("log_scheduler")
      .insert([{ message: `스케줄러 에러 : ${error.message}` }])
  } finally {
    await pgClient.end()
  }
})
```

Serverless function은 사용자의 요청에 응답을 내보낸 후 즉시 프로세스가 종료된다. 

이는 요청 처리가 완료된 후에도 수행되어야 하는 백그라운드 작업의 실행에 제약을 초래해서 결국은 스케줄러가 돌아가지 않게된 것 이다.

### 두번째 방향성
cron이라는 기능을 이용하는 것이였다.

`Cron은 Unix기반 시스템에서 시간 기반의 작업 스케줄러`를 말한다. 사용자가 지정한 시간에 자동으로 스크립트 실행, 명령어 실행 등의 `작업을 반복적으로 수행하도록 예약`할 수 있게 해준다. 

알아보니 gitAction으로도 cron을 사용할 수 있고 버셀로도 cron을 사용할 수 있는 것 같았다.

하지만 버셀 cron은 지금은 기억이 안나지만 그 당시에 제약이 좀 있고 시행착오가 많을 것 으로 예상해 gitAction으로 cron을 구현하기로 했다.

아래는 /workflows/cron.yml 인데 이렇게 시간을 정하면 특정 url을 호출하는 방식으로 진행했다. 

```
name: Call URL every 

on:
  schedule:
    - cron: '0 15 * * *'

jobs:
  call-url:
    runs-on: ubuntu-latest
    steps:
      - name: Call URL
        run: curl -X GET "https://growth-tracker-text.vercel.app{% raw %}${{ secrets.CRON_URL }}{% endraw %}"

```

## ReactQurey를 썼는데 왜 캐싱을 안하셨어요?
라는 질문이 훅 들어왔다. 

핑계 뿐이지만 성능 체크까지 하기에는 시간이 부족하고 캐싱을 해야겠다라는 생각 자체를 까먹고 있었다.

하지만 얼마나 일정관리를 못했으면 이걸 못했을까? 라고 생각할까봐 그리고 시간도 없고 깜빡했다라는 답변을 내놓기에는 뭔가... 내 자신이 너무 초라해보였다.

그래서 `아직 캐싱할 이유를 찾지 못했다` 라고 답변을 했다. 

이 답변에는 많은 의미가 포함된다. 

프로젝트가 많이 바빠서 성능체크를 못했겠구나 또는 아직 실력이 부족하구나라고 넘어갔거나 나라면 이렇게 했을 것 같다는 따뜻한 조언하나 남기고 가시겠지 생각했다. 

하지만... 돌아온 말은 그럼 ReactQurey를 왜써요? 라는 말이였다... 
나도 그런적이 있어서 부끄럽지만 결국은... 누군가를 까내리기위해 질문을 하신거구나 라는 생각을 할 수 밖에 없었던 거 같다. 

이 계기로 한번 더 현업에서 코드리뷰나 커뮤니케이션을 할 때 날카로운 말을 하는 내 자신을 조심해야겠다라는 다짐을 하게 된 거 같다.
