## 기존로직 보존
기존로직이 확장성있게 가져가고 있는 부분도 있고 아닌부분도 있지만 사실상 아닌 부분은 메소드명이나 객체명 등 명칭이 국민은행에 쏠려있다는 것 외에는 괜찮았던거 같다. 

Controller에서 요청도 받고 비지니스로직도 있는 형태인 레거시 코드들이 종종 보였는데 KB 연동 로직들이 그러한 로직들이 존재했다.

우선 controller에 비지니스로직까지 있던 코드들을 Service를 만들어 옮겼고 
엔드포인트에 국민은행의 구분값을 받고 있어서 우리은행 구분값을 만들어 각 구분값에 따라 service를 호출할 수 있도록 `ServiceProvider` 를 만들었다

```java
@Component
@RequiredArgsConstructor
public class BankServiceProvider {

	private final ListableBeanFactory beanFactory;

	public BankService BankServiceProvider(String platform) {
		try {
			return beanFactory.getBean(platform, BankService.class);
		} catch (BeansException ex) {
			throw new RuntimeException(ex);
		}
	}
}

```

`BankService` 는 인터페이스로 각 국민은행과 우리은행 SerivceImpl 구현체를 두었다.  각 구현체에는 구분값을 정의해줘야하는데 `@Service("{구분값}")` 로 구분할 수 있다.

```java
@Slf4j
@Service("국민은행")
@RequiredArgsConstructor
public class 국민은행ServiceImpl implements BankService {...}
```

## 무결성
우리은행에서 고유식별값을 암호화해서 넘겨주는게 이 고유식별값이 우리은행에서 보내준 값이 맞는지 체크하기위해서 RSA 알고리즘을 이용해 비대칭키를 이용하기로 제안을 드렸다. 

처음에는 HmacSHA256로 제안드렸으나 내가 무언가 알고리즘으로 암호화하고 복호화하는 로직을 구현해본적도 없어서 선임님께서 제안해주신 RSA 비대칭키를 이용해 보안과 무결성을 유지하기로 했다. 

openssl로 2048비트의 비공개키를 생성하고 공개키를 추출하는 방향으로 진행했고 비공개키는 우리가 사용하며, 공개키를 우리은행측에 전달했다.

> 비공개키 생성
```
openssl genpkey -algorithm RSA -out private_key.pem -pkeyopt rsa_keygen_bits:2048
```

>  공개키 추출
```
openssl rsa -pubout -in private_key.pem -out public_key.pem
```


## Trancsaction 롤백이 안되네?
회원가입로직이 존재하는데 해당 로직에 우리은행으로 유입된 사람들도 회원이 될 수 있도록 수정하고 예외처리가 잘 되는지 체크하고 있었다.

근데 이게 웬걸? 예외로 의도한 에러가 터지기전 user 테이블에 저장시킨 데이터가 에러가 발생해 중단되어도 롤백이 되지 않는 것이다.

이걸로 야근 1시간 정도를 소요하고 해결하지 못해 다음날 보기로 했다.
항상겪는거지만 잘 안풀리는 문제도 푹쉬고 다음날 보면 민망하게도 바로 캐치하는 경우가 많은데 이번에도 그랬다. 


### 원인...
controller -> service의 A메소드를 호출하지만 해당 메소드는 트랜잭션이 걸려있지 않았다. 

A메소드에서 다른 메소드들을 호출하는데 그 메소드들 중에 user테이블에 데이터를 적재하는 B메소드를 호출한다. 그 메소드에는 트랜잭션이 걸려있었지만 A메소드와 같은 클래스에 존재하기에 트랜잭션이 걸리지 않았던 것이다. 

A메소드에는 B메소드 다음에 C메소드를 호출하는데 이전 개발자의 의도는 C메소드가 실패해도 B메소드를 성공시키고 싶었던거 같다....

우선 A메소드에 트랜잭션을 걸어 하나의 작업단위로 묶어주었지만 추 후에 service의 상위 계층을 두거나 다른 Service만들어 B메소드와 C메소드가 각 다른 트랜잭션에서 돌아갈 수 있도록 보수를 해야할 것 같다. 

아마 좋은 방법은 Service를 만들어 같은 계층을 참조하는 것보단(순환참조...) 상위계층을 두는게 좋다고는 생각이든다.
